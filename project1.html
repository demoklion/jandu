State of the Art Design System for the New Era of Banking

Target 1/3

How can we build a set of sustainable and adaptable components for new daily banking app for 3 platforms‚ÄîiOS, Android, Web‚Äîwithout having the final visual style ready?

My role 2/3

The gist of it

After several unsuccessful attempts by other experts, I led the effort to a successful release. During that, we also defined ways of working and governance of design systems across the corporation.

More details

We started by reviewing prior art. What libraries, nascent or successful design systems exist? We already knew about a component library for the old banking, made in Sketch, to fit with technical documentation in Confluence. We also knew about nascent library for most popular part web banking. During research, we also discovered a few smaller initiatives, for eg. internal apps or people exploring new tools by trying to make libraries.

New sales channels would have a completely new backend and frontend and design stack, though. So we took advantage of that and started defining the minimum useful set of components. Forms, buttons, simple illustrations and templates. These were great artifacts for talking about components with devs, analysts and visual design parts of the effort.

Then we created first mockups of simple flows, that would be in the new channels. Things like login, onboarding, choosing account number, ordering a card. Stuff that we already knew would need to be there. On these real screens, we already iterated on the component setup, parameters, APIs and implementations on all 3 supported platforms (iOS, Android, Angular for web).

During this, we also started to setup a small design system teams to take care of this new internal product. We already had devs and UI designers, we fomalised roles of platform‚Äìspecific design leaders and added a few DS‚Äìspecialist UX designers. Later, we added copywriters and a responsible manager.
